System Prompt · MVP · v2 (battle-tested)

Build a Prediction Market Platform (PredictX)

Next.js 14 · Solidity 0.8.24 · viem/wagmi · Socket.io · Prisma · Docker — env-driven white-label

00How to use this prompt

This is the complete, corrected spec for a Polymarket-style prediction-market MVP. It already bakes in every decision and fix discovered while building & deploying it once — follow it literally and you reproduce a working, deployed app without re-hitting the same traps. Pay special attention to the ⚠ items: each marks a mistake that is easy to make and expensive to undo.

01Goal & constraints

  • Users bet on YES/NO outcomes of future events using on-chain USDC. AMM pricing, owner-resolved outcomes, live updates.
  • TypeScript everywhere, strict. All sensitive config via env. Every on-chain write has error handling + user-friendly messages.
  • Blockchain is the single source of truth. Postgres is a rebuildable cache — it must be reconstructable purely from chain events/state (and the app must self-heal it on startup).
  • MVP scope; built to extend into a multi-tenant white-label protocol.

Contracts: Solidity ^0.8.20 (compile 0.8.24), Hardhat + hardhat-toolbox, ethers v6, OpenZeppelin v5 (Ownable needs Ownable(msg.sender), ERC20 custom errors), TypeChain ethers-v6.

Web: Next.js 14 (App Router) · React 18 · wagmi 2 / viem 2 / RainbowKit · Zustand · Recharts · Socket.io(-client) · Prisma + PostgreSQL · node-cron · sharp · tsx.

04Smart contracts

Three: AMMPricer.sol (internal pure library — inlined, no separate deploy/link), PredictionMarket.sol, MockUSDC.sol (6-dec ERC20 with open mint()/faucet() — testnet only).

AMM pricing — get the direction right

yesPrice = yesPool / (yesPool + noPool)        // a side's OWN pool over the total
noPrice  = noPool  / (yesPool + noPool)
shares   = usdcAmount / price = usdcAmount * total / ownPool
invariant: yesPrice + noPrice == 1.0  (±1 wei rounding)
⚠ Buying a side RAISES that side's price (adds to its own pool → its price up, the other down) — like a real prediction market. Do NOT use price = oppositePool/total (that inverts it: buying YES would lower YES%). getShares uses the own-side pool. Keep the Solidity library and the frontend JS replica (lib/amm.ts) in lock-step.

PredictionMarket.sol

  • State: Market{ id, question, endTime, resolved, outcome, yesPool, noPool, totalShares }; mappings yesShares/noShares[id][addr], totalYes/NoShares[id]; usdc (immutable), treasury, feeBps, accumulatedFees, marketCount.
  • constructor(usdc, treasury, feeBps) — owner = deployer; require non-zero usdc/treasury, feeBps ≤ 500.
  • createMarket(question, endTime, seedAmount) onlyOwner — seeds both pools (seedAmount/2 each, odd wei to NO); pulls seed USDC; emits MarketCreated. Seed mints no shares (it subsidises traders).
  • buyShares(id, isYes, usdcAmount, minShares) — require not resolved, now < endTime, amount>0; fee = usdcAmount*feeBps/10000 kept for treasury (NOT in pool); net enters pool; shares = AMMPricer.getShares(...); require shares ≥ minShares (slippage); pull full usdcAmount; nonReentrant; emit SharesBought(id, buyer, isYes, usdcAmount, shares, newYesPool, newNoPool).
  • resolveMarket(id, outcome) onlyOwner — require expired, not resolved.
  • claimWinnings(id) — payout = userWinningShares / totalWinningShares * totalPool; zero the claimer's winning shares (prevents double claim); pool & totals untouched so remaining winners keep proportions; nonReentrant; pull model, claimable forever, no admin withdrawal of unclaimed funds.
  • Fees: setFeeBps(≤500) onlyOwner, setTreasury(non-zero) onlyOwner, withdrawFees() treasury-only (require >0).
  • Views: getMarket, getShares(addr,id), getYesPrice/getNoPrice (1e18), getTotalPool.

Events (backend listens to these)

MarketCreated(uint indexed id, string question, uint endTime, uint seedAmount)
SharesBought(uint indexed id, address indexed buyer, bool isYes, uint usdcAmount, uint shares, uint newYesPool, uint newNoPool)
MarketResolved(uint indexed id, bool outcome)
WinningsClaimed(uint indexed id, address indexed claimer, uint usdcAmount)

Tests (Hardhat, ~35, must pass)

Use an AMMPricerHarness.sol to expose the internal library. Cover prices/invariant/extreme ratios; shares at 50/50 and imbalanced; createMarket happy + reverts; buyShares happy + fee + feeBps=0 + every revert (expired/resolved/zero/slippage/no-allowance via revertedWithCustomError(usdc,'ERC20InsufficientAllowance')); resolve YES/NO + reverts; claim pro-rata (read share counts back from chain — don't hardcode if you might tweak the formula) + loser/double-claim revert; fee mgmt; full lifecycle integration with balance reconciliation.

05Slippage & contract error handling

  • Frontend computes minShares = expectedShares * 0.99 (1% tolerance) and passes it in.
  • Show a price-impact warning when |priceAfter-priceBefore|/priceBefore > 5%.
  • Map contract errors to friendly text: slippage→"Price moved too much", market expired→"This market has closed", already resolved, no winning shares, user rejected (4001), insufficient allowance/balance, fallback "Transaction failed: " + shortMessage. Parse nested viem shortMessage/details/cause.

06Database (Prisma · cache layer only)

Models: User(address @id, role, createdAt), Category(id, name@unique, slug@unique, icon, isDefault), Market, Trade(id, marketId, trader, outcome, amount Float, shares Float, txHash @unique [idempotent replay], createdAt).

Market = id (String = String(contractId)), contractId @unique Int, question, description, categoryId→Category, endDate, resolved, outcome?, status ("ACTIVE"|"EXPIRED"|"RESOLVED"), imageUrl?, yesLabel, noLabel, yesImageUrl?, noImageUrl?, cached chain state: yesPool/noPool/volume Float, createdAt. Index status, categoryId, endDate.

⚠ The original spec's Market model has no pool/volume columns, but the listing + sort-by-volume need them. Add cached yesPool/noPool/volume (updated by the event listener); derive prices from pools in a serializer. Prices are never stored.
Money: expose USDC to the UI as a plain human number; convert at the chain boundary only (6-dec ints ↔ number). USDC is 6 decimals on every chain — assumption holds.

07Data strategy & self-healing (critical)

Chain holds everything (markets, pools, shares, trades-as-events, outcomes). The DB mirrors it for fast queries + off-chain metadata (description, category, images, labels — these are NOT on chain).

  • Event listener (viem watchContractEvent): SharesBought → update pools/volume, upsert Trade (by txHash), upsert User, emit market:trade. MarketResolved → update + emit market:resolved. MarketCreated → ensure a row exists (UI-created rows already carry metadata, so don't overwrite).
  • On every startup, reconcile from chain: read marketCount; for each market, update existing rows' chain-derived fields and RE-CREATE any missing rows (with on-chain question + default metadata). Then backfill trades from SharesBought logs (chunked getLogs, upsert by txHash) and recompute volume from the Trade table.
⚠ This self-heal is mandatory, not optional. Free hosts spin down / wipe ephemeral state and deploys restart the process — without startup reconcile+backfill, markets and trades silently vanish from the UI. Metadata that is only off-chain (descriptions, custom labels, images) can't be recovered from chain, so never wipe the DB casually.

09Realtime (Socket.io on a custom server)

server.ts runs Next + Socket.io + cron + event listener in one process. Rooms: market:{id}, market:list, admin (admins only, after signature verify with viem verifyMessage). Guests connect unauthenticated to public rooms.

⚠ Share the Socket.io instance via globalThis, not a module singleton: server.ts (run by tsx) and the Next-bundled API routes (webpack) are different bundles, so a plain module-level singleton diverges and API-route emits silently no-op.

Payloads (in shared): market:trade (→ market:{id} + market:list), market:resolved (→ market:{id}), market:created (→ market:list), market:expired (→ both), admin:resolve:pending (→ admin). Client connects silently after wallet auth; sign "Connect to PredictX" once (cache the signature in sessionStorage). Never auto-open the connect modal.

10Cron + client-side expiry

node-cron '* * * * *': ACTIVE markets with endDate ≤ now → EXPIRED, emit market:expired + admin:resolve:pending.

⚠ The cron only fires on the minute, so server status lags expiry by up to ~60s. Derive an "effective status" on the client (a market is EXPIRED the instant endDate passes) with a hook that forces a re-render at the exact end time. Cards/detail/trading panel use the effective status so a market closes the moment the countdown hits zero.

11Pages & features

  • Listing (/): status tabs (Open default / Resolved / All — resolved hidden by default so they don't clutter), category pills, sort. Cards: image/emoji, question (2-line clamp), category pill, status badge, combined YES/NO probability bar, animated volume, time remaining. Live-updates via socket.
  • Market detail (/market/[id]): two columns. Left: big YES/NO price blocks, price chart (Recharts, crosshair + active dot, 1D/1W/1M), collapsible description, activity feed (gradient avatars, last 50, live prepend). Right sticky: TradingPanel + market-info card. Admin sees an Edit link.
  • TradingPanel — ACTIVE: segmented YES/NO toggle (sliding indicator via framer-motion layoutId, optional outcome images), amount + MAX, live preview (shares/price/fee[hidden if feeBps=0]/payout), price-impact warning. Button flow: connect → Approve USDC (MaxUint256, once) → Buy → Confirmed. EXPIRED: "Closed · awaiting resolution", read-only. RESOLVED: ResolutionBanner + Claim only if the wallet actually holds winning shares (read getShares; otherwise "you didn't win"/"no position").
  • Portfolio (/portfolio): Trader+; summary cards (Open / Unrealized P&L / Claimable); Open/Resolved tabs; claim button per winning position. On connect, toast if any unclaimed winnings.
  • Create (/create, admin): form + live card preview. Fields: question 10–140, description 20–1000, category, endDate+endTime (≥1h future, UTC), seed 10–10000 (≤ balance), market image, yes/no labels (both or neither), optional per-outcome images. Flow: upload image → approve USDC → createMarket → POST metadata → redirect.
  • Edit (/market/[id]/edit, admin): see §13.
  • Admin Resolve (/admin/resolve): expired+unresolved oldest-first, confirm modal, badge count, realtime.
  • Admin Categories (/admin/categories): list + counts + Default badge; add (emoji+name+slug auto); delete custom.
  • Admin Fees (/admin/fees): fee rate, accumulated fees, Withdraw fees (treasury) and Set fee (owner, 0–5%).
  • Skeletons, toasts (pending→confirmed→error), empty states, error boundaries per page, 404 for unknown ids.

12Images — store as data URIs (no filesystem)

⚠ Do not write uploads to public/uploads. On ephemeral hosts (Render free) the disk is wiped on restart and runtime-written public files don't serve reliably → broken images. Instead: POST /api/upload resizes with sharp (≈256×256 webp) and returns a base64 data URI stored directly in the DB imageUrl. Fall back to the original bytes if sharp ever fails. Same for per-outcome images. (For huge volume, swap to S3/R2 later — then imageUrl becomes a real URL.)

Per-outcome images (yesImageUrl/noImageUrl) are purely off-chain — no contract change. Shown in the probability bar (replacing the dot), the trading toggle, detail price blocks, and cards.

13Editing markets — integrity locks

Admin can edit off-chain metadata, but a market's terms must not change under traders:

FieldRule
questionNever editable — it's the on-chain commitment (immutable). Shown read-only.
description, yes/no labelsEditable only before any trade. Once totalShares > 0 on-chain → locked (they define the bet/criteria).
category, all imagesAlways editable — cosmetic, don't change the bet.

Enforce on the server (PATCH reads on-chain totalShares; returns 409 on a locked-field change) AND reflect in the UI (disabled inputs + a "trading has started" note). Server is authoritative.

15White-label & runtime config

One codebase, configured by env only. Branding via TENANT_* → injected as CSS variables on <html> at runtime; components reference tokens (text-yes, bg-brand), never raw hex. Per-tenant: CONTRACT/ADMIN/TREASURY/FEE_BPS, DATABASE_URL, RPC, chain id.

⚠ Resolve NEXT_PUBLIC_* at RUNTIME, not build time. Some hosts (Render) don't pass env to the Docker build, so build-time-inlined values come out empty. Have the root layout (server, force-dynamic) read process.env at request time and inject window.__PREDICTX_ENV__ via an inline script BEFORE the app bundles run; lib/constants reads that on the client, with a process.env fallback on the server. Bonus: changing config = restart, no rebuild.

Future protocol direction (not MVP): a Factory contract that lets clients deploy their own market contract (they pay gas, they own it) with an immutable protocol fee → your address on every trade; one static multi-tenant frontend reading everything from chain + IPFS — so a tenant self-serves and you just collect the fee.

18Deployment (Docker → free host now, VPS later)

Single Dockerfile (node:22-bookworm-slim): install all workspace deps, npm run compile (ABIs) → prisma generate → next build; entrypoint: prisma db push → (optional RESET_DB wipe) → seed categories → start server.ts. docker-compose.yml (app + Postgres) for any VPS.

  • Recommended free stack: Render (Docker Web Service) + Render/Neon Postgres + a public testnet (Base Sepolia / Polygon Amoy). Render free spins down after ~15 min — the startup self-heal (§07) covers it.
  • Flow: deploy contracts to the testnet (npm run deploy:<net>, SEED_MARKETS=false) → push repo → connect on Render with Docker → set env vars → deploy.
⚠ Prisma on the slim image: install openssl ca-certificates in the Dockerfile and set binaryTargets = ["native","debian-openssl-3.0.x"] in the schema, or db push fails with "Schema engine error".
⚠ RESET_DB is one-shot. Set it true for a single deploy after a contract redeploy, then REMOVE it — otherwise every spin-down/restart wipes admin-created markets.

Switching chains is env-only (+ ~2 lines to add a brand-new chain to lib/chains.ts and hardhat.config.ts): redeploy contracts, update CHAIN_ID/RPC/addresses (use the chain's real USDC on mainnet, MockUSDC on testnet), reset DB. App code never changes.
